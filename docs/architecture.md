# qrlib 架构说明

## 仓库定位

`qrlib` 是一个面向三维深度学习基础能力的可复用代码库，不用于承载完整实验工程。
这个仓库应该保持边界清晰、公共接口稳定，并且方便被其他项目直接依赖。

## 模块边界

- `qrlib.data`：采样、`collate`、数据增强、格式转换等通用数据处理代码
- `qrlib.geometry`：点云、网格、几何归一化等稳定几何接口
- `qrlib.metrics`：可复用的指标计算代码

这些子包是按“稳定领域能力”组织的示例，而不是固定上限。只要边界清晰、接口稳定，就可以继续增加新的子包。

## 代码迁入规则

只有同时满足下面几个条件时，才应把代码迁入 `qrlib`：

1. 这段代码已经在至少两个项目中复用
2. 它的接口已经足够稳定，适合被其他项目依赖
3. 它自然属于一个边界清晰、文档化的稳定子包

如果一段代码仍然偏实验性、只服务某篇论文、或强依赖某个数据集与训练流程，就先留在具体项目仓库中。

## 需要避免的情况

- 添加泛化但边界不清的 `utils.py`、`helpers.py`、`common.py`
- 在 `qrlib.__init__` 中一次性导出大量内容
- 在仓库中存放数据集、权重、实验日志、notebook 输出
- 在代码尚未稳定时就复制完整模型或完整流程进来
- 在没有真实需求时提前创建大量空子包

## 公共接口建议

建议使用者显式从子包导入：

```python
from qrlib.data import ...
from qrlib.geometry import ...
from qrlib.metrics import ...
```

除非某个内部路径已经被明确提升为文档化的公共接口，否则不要依赖过深的内部实现路径。

## 文档、示例与测试的组织原则

- 文档按公开能力组织，统一收敛到 `docs/`，不要在源码旁重复维护一份面向用户的说明
- 示例按使用场景组织，同类能力可以共享一个完整示例，不要求每个源码文件都单独配一个例子
- 测试按行为风险组织，重点覆盖公开 API、边界条件、形状约束、数值计算与兼容路径
- 内部实现、私有辅助函数或薄包装层不要求单独提供用户文档或示例，但其关键行为仍应被测试直接或间接覆盖

以 `qrlib.geometry` 为例，用户文档集中放在 `docs/geometry/`，示例集中放在一个 geometry 工作流示例里，而测试继续拆分为类型校验、归一化行为和 torch 兼容性三类。
